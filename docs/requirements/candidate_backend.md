# 后端方向考题：工作项状态流转与 AI 需求分析服务

## 1. 考题目标

请基于通用业务场景，实现一个后端服务，用于管理研发工作项、处理状态流转、维护澄清问题，并提供一个 AI 辅助分析能力。

本题重点考察：

1. 后端领域建模能力。
2. 状态流转和业务规则设计能力。
3. API 设计能力。
4. 数据持久化与错误处理能力。
5. 基础测试能力。
6. AI 能力工程化封装能力。
7. 借助 AI 完成分析、编码、测试和文档的能力。
8. 借助 AI 或简单实现补齐基础前端演示能力。

## 2. 技术要求

技术栈：

- Java / Spring Boot

数据库可使用：

- H2 / SQLite / PostgreSQL / MySQL

请在 README 中说明选择理由。

## 3. 必做功能

### 3.1 工作项管理

实现工作项的创建、查询、修改等基本能力。

题目不指定具体 API 路径、请求结构和响应结构。你需要自行设计 API，并在文档中说明：

1. API 如何划分。
2. 请求和响应如何设计。
3. 错误如何表达。
4. 是否采用 REST、RPC 或其他风格。
5. 为什么这样设计。

### 3.2 状态流转

实现工作项状态流转能力。

要求：

1. 明确定义状态集合。
2. 明确定义合法流转规则。
3. 非法流转必须被拦截。
4. 状态变化应记录历史。
5. 完成状态是否允许变更，需要说明规则。
6. 状态流转应有清晰错误返回。

### 3.3 澄清问题管理

实现工作项澄清问题的维护能力。

要求：

1. 支持新增澄清问题。
2. 支持回复或解决澄清问题。
3. 支持查询工作项下的澄清问题。
4. 澄清问题应包含严重程度和解决状态。
5. 高优先级未解决澄清问题应影响状态流转。

### 3.4 核心业务规则

至少实现以下规则：

```text
如果工作项存在未解决的高优先级澄清问题，则不允许进入后续可开发状态。
```

你可以自行定义“高优先级”和“后续可开发状态”的具体含义，但需要在文档中说明。

### 3.5 AI 辅助分析服务

实现一个 AI 辅助分析能力。

至少支持以下一种能力：

1. 生成需求摘要。
2. 生成验收标准建议。
3. 识别风险点。
4. 生成澄清问题。
5. 生成任务拆解建议。

要求：

1. AI 能力应通过清晰的服务或模块封装。
2. 可以调用真实 LLM，也可以使用 Mock 实现。
3. 如果使用 Mock，需要说明如何替换为真实 AI 服务。
4. AI 分析结果应结构化返回，而不是只返回一段散文。

### 3.6 简单前端页面（必做，可简化）

后端方向也需要提供一个可运行的简单前端页面，用于演示核心流程。

要求：

1. 页面可以很简单，不要求看板、不要求精美 UI。
2. 至少支持查看工作项列表或工作项详情。
3. 至少支持触发一次状态流转。
4. 至少支持查看或新增澄清问题。
5. 至少支持触发 AI 分析并展示结构化结果。
6. 可以使用原生 HTML/JS、简单 React/Vue 页面、服务端模板、Swagger UI 之外的自定义调试页面，或由 AI Agent 生成的简易页面。

### 3.7 测试

至少提供核心业务规则测试。

建议覆盖：

1. 合法状态流转。
2. 非法状态流转。
3. 高优先级澄清问题阻断状态流转。
4. AI 分析服务的基本行为。

## 4. 加分功能

以下内容不是必须，但会作为加分项：

1. OpenAPI / Swagger 文档。
2. 状态流转历史查询。
3. 并发更新保护或乐观锁。
4. 统一错误码体系。
5. 简单认证或用户上下文。
6. Docker Compose。
7. 更完整的前端页面，例如简单看板、详情页或状态历史展示。
8. 更完整的测试覆盖。
9. 使用 AI Agent 生成过程文档或开发记录。

## 5. 不强制要求

后端方向不强制要求：

1. 实现完整前端看板或复杂前端工程。
2. 实现复杂登录权限体系。
3. 真实调用 LLM。
4. 完整企业级数据模型。

但需要说明：

1. 当前简单前端页面如何调用后端能力。
2. 如果由专业前端扩展为完整应用，需要哪些 API 和状态字段。
3. AI 如果要真实接入，当前设计如何扩展。
4. 如果要支持多人协作或权限控制，设计上如何演进。

## 6. 交付建议

建议优先保证以下闭环：

```text
打开简单前端页面 → 创建或查看工作项 → 添加澄清问题 → 尝试状态流转被阻断 → 解决澄清问题 → 状态流转成功 → 触发 AI 分析 → 展示结构化分析结果
```

完成后再考虑加分项。
