# 通用业务场景：AI 辅助研发工作项流转与需求澄清系统

## 1. 背景

你所在团队正在建设一个软件工程平台，用于帮助研发团队管理工作项、澄清需求、跟踪状态流转，并通过 AI 辅助识别需求风险、生成验收建议或澄清问题。

现在需要实现一个轻量级模块：**AI 辅助研发工作项流转与需求澄清系统**。

该模块面向研发团队内部使用，目标是支持：

1. 管理研发工作项。
2. 跟踪工作项从需求分析到完成的状态流转。
3. 记录需求澄清问题及其处理结果。
4. 提供 AI 辅助分析能力，帮助研发人员识别需求风险、生成验收标准建议或澄清问题。
5. 支持基本演示和验证。

## 2. 工作项基本概念

系统中存在“工作项”这一核心对象。工作项可以是需求、缺陷或任务。

工作项应至少包含以下业务信息，但候选人可以根据自己的设计进行增减：

| 信息 | 说明 |
|---|---|
| 标题 | 工作项名称 |
| 描述 | 需求、缺陷或任务描述 |
| 类型 | story / bug / task 等 |
| 优先级 | 高、中、低或 P0/P1/P2 等 |
| 状态 | 工作项当前所处流程阶段 |
| 负责人 | 当前处理人 |
| 标签 | 用于分类或搜索 |
| 验收标准 | 判断工作项是否完成的条件 |
| 风险等级 | 候选人可自行设计 |
| 澄清问题 | 与需求不明确处相关的问题和答案 |
| 状态历史 | 状态变化记录 |

> 注意：题目不规定具体数据库结构、具体 API 路径或具体前端组件结构，候选人需要自行设计并说明理由。

## 3. 状态流转要求

系统应具备明确的工作项状态流转机制。候选人可以自行命名状态，但至少需要覆盖以下含义：

```text
草稿 / 待分析 / 已准备 / 开发中 / 测试中 / 已完成
```

需要考虑：

1. 哪些状态可以流转到哪些状态。
2. 哪些流转属于非法流转。
3. 是否允许退回。
4. 完成状态是否允许再次变更。
5. 状态变化是否需要记录历史。
6. 需求澄清问题是否会影响状态流转。

建议业务规则：

> 如果工作项存在未解决的高优先级澄清问题，则不允许进入“已准备”或“开发中”等后续状态。

候选人可以自行扩展规则，但需要在文档中说明。

## 4. 澄清问题要求

系统应支持为工作项记录澄清问题，例如：

- 需求边界不清。
- 验收标准缺失。
- 异常流程未定义。
- 权限或数据范围不明确。
- 接口返回字段不明确。

澄清问题应至少包含：

| 信息 | 说明 |
|---|---|
| 问题内容 | 需要澄清的问题 |
| 严重程度 | 高/中/低或 P0/P1/P2 |
| 状态 | 未解决 / 已解决 |
| 回答 | 对问题的回复 |

## 5. AI 辅助能力要求

系统需要提供一个简单 AI 辅助能力。可以实现以下任意一种或多种：

1. 根据工作项描述生成需求摘要。
2. 根据工作项描述生成验收标准建议。
3. 根据工作项描述识别风险点。
4. 根据工作项描述生成澄清问题。
5. 根据工作项内容生成任务拆解建议。

AI 能力可以采用以下任意方式实现：

| 方式 | 是否接受 | 说明 |
|---|---|---|
| 调用真实 LLM API | 接受，加分 | 可使用 OpenAI-compatible、本地模型或其他服务 |
| Mock AI Service | 接受 | 通过规则或固定模拟返回结构化结果 |
| 前端本地模拟 | 前端方向接受 | 需说明如何替换成真实服务 |

重点考察：

```text
候选人是否能将 AI 能力作为工程能力进行封装，而不是简单拼接字符串或写死结果。
```
